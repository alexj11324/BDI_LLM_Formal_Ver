## Task 5: BDI Engine (`src/core/bdi_engine.py`) — COMPLETED (2026-03-02)

Implemented `BDIEngine` class in `src/core/bdi_engine.py`:

- Constructor accepts `BaseDomainVerifier` and a DSPy teacher module (dependency injection via `TeacherModule` Protocol).
- `extract_json_dag(raw_response: str) -> dict` static method: regex-based extraction (first `{` to last `}`) with `json.loads` fallback. Raises `ValueError` on failure (treated as verification failure).
- Full main loop logic:
  1. Pops desire from FIFO queue (`deque`).
  2. Prompts Teacher LLM via `_invoke_teacher()` with belief state, desire, and domain context.
  3. Passes deep copy of BeliefState + IntentionDAG to `self.verifier.verify_transition()`.
  4. On success: executes DAG (optional `dag_executor`), updates BeliefState, logs `GoldenTrajectory`.
  5. On failure: injects error traces + correction hints into context, increments retry counter.
- **Epistemic Deadlock Guard**: If `retries > MAX_RETRIES` (default 3), raises `EpistemicDeadlockError`. Catches it, suspends task to `BeliefState.suspended_intentions`, compresses failed traces into epistemic memory flags, invokes recovery teacher.
- **Zero Domain Leakage**: NO domain-specific logic. Only imports generic types, Pydantic schemas, and `BaseDomainVerifier`.
- `GoldenTrajectory` class for recording successful BDI reasoning traces.
- `_hydrate_dag()` method with sensible defaults for missing top-level keys.

Verified: `python -c "from src.core.bdi_engine import BDIEngine"` — OK.

## Task 9: DSPy Signatures (`src/dspy_pipeline/signatures.py`) — COMPLETED (2026-03-02)

Implemented three DSPy Signatures that force structured BDI reasoning:

- `BDIReasoningSignature(dspy.Signature)`: Primary generation path.
  - Inputs: `belief_state` (str), `desire` (str), `domain_context` (str).
  - Outputs: `reasoning` (str), `intention_dag_json` (str).
  - Docstring instructs LLM to reason about preconditions/effects and produce valid IntentionDAG JSON.
- `ErrorCorrectionSignature(dspy.Signature)`: Retry/correction path.
  - Inputs: `belief_state` (str), `failed_dag` (str), `error_trace` (str), `correction_hint` (str).
  - Outputs: `reasoning` (str), `corrected_dag_json` (str).
  - Docstring instructs LLM to analyse specific node failures and produce a corrected DAG.
- `RecoveryPlanSignature(dspy.Signature)`: Epistemic deadlock recovery.
  - Inputs: `suspended_intentions` (str), `epistemic_flags` (str).
  - Outputs: `reasoning` (str), `recovery_plan_json` (str).
  - Docstring instructs LLM to propose alternative strategies/sub-goal decomposition.

Design decisions:
- All field types are `str` (serialised JSON) because DSPy Signatures operate at the text level.
- Signature docstrings serve as system instructions (DSPy convention).
- No domain-specific logic — fully domain-agnostic by design.
- All fields use `dspy.InputField()` / `dspy.OutputField()` with detailed `desc` parameters.
- Comprehensive module-level docstring documenting all three signatures.
